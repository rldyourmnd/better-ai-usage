# V1.0.0 Discovery Questionnaire (for final architecture lock)

Заполни, пожалуйста, максимально конкретно. Чем точнее ответы, тем меньше рисков при реализации.

## Ответы на v1.0.0 (предварительные, финальные для фиксации)

1) Основной пользователь: продуктовый/визуальный владелец AI-ориентированного терминала (любой, кто много работает в CLI и хочет стабильную и быструю работу с длинными сессиями), с фокусом на fish + starship UX.
7) Сценарии по shell из коробки: дефолт `fish + Starship`, разрешен безопасный fallback `zsh`; Bash не является базовым целевым shell в v1.0 на Linux/macOS.
8) Нужна поддержка совместимости с bash-образными/псевдо-терминал-поведенческими нормами: приоритет — корректность и стабильность для критичных сценариев (цвета, курсор, стили текста, очистка экрана, управление курсором, scroll и paste), при этом производительность/чистота рендера выше «редких/экзотических» legacy-расширений ANSI.

1) Целевые профили пользователей: кто основной пользователь и через какой CLI чаще всего запускается терминал?
2) Какие типы задач приоритетны на старте: только ручной запуск процессов, AI-пайплайны, длинные фоновые сессии, или mix?
3) Какой SLA по падениям по твоему мнению приемлем на v1.0: ноль падений за сессию, только деградация GPU->CPU, или допустимы разовые restartы сервиса?
4) Какой уровень автоматического фолбэка ты ожидаешь: мгновенный авто-switch на CPU, или сначала уведомление + подтверждение пользователя?
5) Максимально допустимый RAM пика для базового окна на Linux/macos в idle и под нагрузкой (примерные пороги).
6) Какой целевой диапазон FPS для обычной работы и для AI-потока с большим stdout потоком?
7) Какие shell сценарии обязательны из коробки: только fish + Starship или нужно сразу гибко поддержать bash/zsh fallback без вмешательств?
8) Нужен ли режим "чистая совместимость с legacy bash-поведенческим ANSI" или приоритет на скорость/согласованность рендера даже с небольшими несовместимостями?
9) Какие события в терминале считаются критичными для AI-инструментов: prompt latency, copy/paste latency, no color bleed, stable scrollback?
10) Какой минимальный размер scrollback ты считаешь обязательным на v1 (строки), и нужен ли лимит для контроля памяти?
11) Требования к мульти-оконности в v1: запрещаем до стабильного single-pane или уже сейчас нужен минимум 2 вкладки/панели?
12) UI-стиль: хочешь ли ты строгий минимализм с куберпанк-палитрой, или гибрид cyber/cold-digital + анимации и motion-спектры?
13) Какая конфигурация GPU допустима: Vulkan-only на Linux или нужно поддерживать Metal и fallback на OpenGL/ANGLE?
14) Нужна ли аппаратная акселерация текстовых эффектов (blur/shadow/gradient) или пока только быстрый «чистый» текст + базовая заливка?
15) Для Windows как приоритет: только архитектурная заготовка (без активной поддержки) или функциональная совместимость на этапе v1.0 beta?
16) Какие механики наблюдаемости нужны из коробки: structured logs, frame time, memory trend, event-id для сбоев GPU?
17) Какие метрики попадут в release-критерии: время старта процесса, latency input->render, p95 frame time, CPU/GPU usage, crash rate?
18) Какой путь обновлений ты хочешь считать приоритетным: apt-only, homebrew-only, или GitHub releases + installer скрипт?
19) Нужны ли профили производительности по профилям CPU-only, GPU-only и смешанному автосценарию на CI?
20) Что считаем багом blocker для релиза: любая деградация старта shell, мерцания, memory leaks, или только crash-и?
21) Какие расширения в roadmap важны уже в дизайне: session recorder, clipboard image/selection, поиск/replace в scrollback, split-screen, devtools-панель?
22) Какой формат документации приемлем для CI-проверок: markdown-спеки с чёткими acceptance criteria или user stories?

1) План интеграции по слоям без противоречий
2) Матрицу рисков и мер по стабильности
